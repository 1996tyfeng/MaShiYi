## 0. 前言

《马十一》不是一个普通的宠物记录项目。

它是一份关于一只小橘猫一生的长期记录。

这个项目开始于一只约两周龄的小奶猫被妈妈带回家的那一天。那一天是农历五月十一，它有了名字：**马十一**。名字里的“十一”，也与“拾遗”同音，寓意拾起一个遗落的小生命。

本项目优先记录真实，而不是戏剧性；优先记录长期变化，而不是短暂热闹；优先保留未来值得回看的细节，而不是堆砌漂亮句子。

很多年以后，当妈妈重新打开这个仓库，希望能清楚地看到：

> 那只小小的、会闻着斗篷味道睡着的小橘猫，是怎样一点点长大的。

---

## 1. 项目愿景

本项目的长期愿景是：

> 用 Markdown、Git 历史和持续记录，为马十一保存一份可以陪伴它一生的成长档案。

它可以是：

- 一份日记；
- 一份健康记录；
- 一份相册索引；
- 一份年度纪念册素材；
- 一份妈妈与小十一共同生活的时间胶囊。

如果未来导出为 PDF、打印成册，书名可以简单叫：

> **《马十一》**

首页固定文案：

> 欢迎回家，小十一。  
> 从今天开始，你姓马。

## 5. 时间与 Day 编号规则

Project MaShiYi 的所有日期、Day 编号、CSV 日期和 Diary 日期，统一以**妈妈发送消息时的北京时间（UTC+8）**为准。

当 AI 本地时间、系统时间或 UTC 日期与妈妈时间冲突时，以妈妈时间优先。

Day 编号不手工猜测，统一按固定公式计算：

```text
Day = 记录日期 - 2026-06-26
```

固定示例：

- 2026-06-26 = Day 0
- 2026-06-27 = Day 1
- 2026-06-28 = Day 2
- 2026-06-29 = Day 3

如果日期和 Day 编号发生冲突，以日期为准，重新计算 Day。

## 6. 更新原则

### 6.1 永远追加

成长记录原则上只追加，不覆盖。

历史 Day 不应随意重写。

如果必须修正错误，应在同一天记录中增加“修正说明”，而不是删除原文。

### 6.2 真实优先

所有故事必须来自妈妈实际描述、照片、视频或当天真实事件。

可以整理语言，但不能虚构事件。

可以温柔表达，但不能为了煽情制造不存在的情节。

### 6.3 数据与故事并重

每天记录应尽量包含：

- 日期；
- Day 编号；
- 体重；
- 奶量或饮食；
- 排尿；
- 排便；
- 精神状态；
- 今日故事；
- 今日观察；
- 今日成就。

如果某项未知，可以写“未记录”“待补充”或 `unknown`。

不要为了完整性编造数据。

### 6.4 数据实时，日记汇总

体重、奶量和排泄可以实时更新进 CSV。

当妈妈明确说“更新今天日记”时，再把当天零散事件整理进 `diary/DIARY.md`。

推荐工作流：

- 实时数据：更新 `data/weight.csv`、`data/milk.csv`、必要时更新 `data/health.csv`。
- 今日日记：更新 `diary/DIARY.md`，必要时同步 `ACHIEVEMENTS.md`、`MEDICAL.md`、`CHANGELOG.md`。

## 9. 数据记录规范

### 9.1 weight.csv

字段：

```csv
date,day,weight_g,measurement_time,notes
```

说明：

- `date`：日期，格式 YYYY-MM-DD，以北京时间为准；
- `day`：Day 编号，由日期计算；
- `weight_g`：体重，单位克；
- `measurement_time`：称重时间，如 `morning`、`before_feeding`、`after_feeding`、`unknown`；
- `notes`：备注。

### 9.2 milk.csv

字段：

```csv
date,day,time,amount_ml,pee,poop,after_status,notes
```

说明：

- `date`：日期，格式 YYYY-MM-DD，以北京时间为准；
- `day`：Day 编号，由日期计算；
- `time`：喂奶时间，如 `08:00`、`evening`、`later`；
- `amount_ml`：奶量，单位 ml；
- `pee`：该顿奶后是否排尿，推荐值 `yes` / `no` / `unknown`；
- `poop`：该顿奶后是否排便，推荐值 `yes` / `no` / `unknown`；
- `after_status`：喝完后的状态，如 `unknown`、`relaxed`、`seeking_milk`、`fast_finished`、`sleepy`；
- `notes`：备注。

奶猫阶段每顿记录比只记总量更有价值。`after_status` 用于记录喝完后的即时状态，帮助未来判断喂养节奏。

### 9.3 health.csv

字段：

```csv
date,day,pee,poop,energy,notes
```

说明：

- `health.csv` 记录每日健康摘要；
- `milk.csv` 记录每顿奶后的排泄细节；
- 二者可以同时存在，但不要互相矛盾；
- 如果当天没有明确健康摘要，可以暂不追加，不要编造。

### 9.4 medical.md / MEDICAL.md

医疗、驱虫、疫苗、就医、用药等长期记录应写入：

```text
diary/MEDICAL.md
```

医疗记录必须客观，不夸张，不自行替代兽医诊断。

## 10. 医疗与健康记录原则

本项目可以记录健康观察，但不能把 AI 观察写成正式诊断。

推荐写法：

- “观察到……”
- “目前表现为……”
- “建议继续观察……”
- “如持续出现，应咨询兽医……”

避免写：

- “确定没事”；
- “一定是……”；
- “不用看医生”；
- “这是病 / 不是病”的绝对判断。

奶猫阶段重点关注：

- 体重趋势；
- 食欲；
- 排尿；
- 排便；
- 精神状态；
- 体温与保暖；
- 呼吸状态；
- 是否呛奶；
- 是否腹泻。

## 15. AI 协作规范

### 15.1 ChatGPT 职责

ChatGPT 负责：

- 整理妈妈提供的每日信息；
- 写成长日记；
- 进行行为观察；
- 提醒健康风险；
- 更新 Markdown；
- 维护 CSV；
- 保持文风统一。

ChatGPT 不应：

- 虚构当天没有发生的事情；
- 擅自改写历史 Day；
- 把不确定医学判断写成确定诊断；
- 忽略妈妈明确提供的事实；
- 在未确认北京时间日期的情况下手工猜 Day 编号。

### 15.2 Codex 职责

Codex 如参与，主要负责：

- 文件结构维护；
- 自动化脚本；
- CSV 处理；
- PDF 导出；
- GitHub Actions；
- 格式检查。

Codex 不应主动改写成长日记文风。

### 15.3 未来其他 AI

任何未来 AI 接手本项目时，必须先阅读：

1. `PROJECT_BIBLE.md`
2. `README.md`
3. `LORE.md`
4. `diary/DIARY.md`
5. `data/*.csv`
6. `prompts/*.md`

再进行任何更新。

- 随意删除历史记录；
