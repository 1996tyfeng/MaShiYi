# Project Bible

> 项目名称：**《马十一》**  
> 仓库：`1996tyfeng/MaShiYi`  
> 建立时间：2026-06  
> 项目性质：长期成长档案 / 数字日记 / 未来纪念册

---

## 0. 前言

《马十一》不是一个普通的宠物记录项目。

它是一份关于一只小橘猫一生的长期记录。

这个项目开始于一只约两周龄的小奶猫被妈妈带回家的那一天。那一天是农历五月十一，它有了名字：**马十一**。名字里的“十一”，也与“拾遗”同音，寓意拾起一个遗落的小生命。

这个仓库要记录的，不只是体重、奶量、排尿和排便。

它还要记录：

- 第一次安心睡着；
- 第一次踩奶；
- 第一次爬到妈妈头边睡觉；
- 第一次表达“我还没吃饱”；
- 第一次健康长大一点点；
- 以及未来很多很多次平凡但珍贵的日子。

本项目优先记录真实，而不是戏剧性；优先记录长期变化，而不是短暂热闹；优先保留未来值得回看的细节，而不是堆砌漂亮句子。

很多年以后，当妈妈重新打开这个仓库，希望能清楚地看到：

> 那只小小的、会闻着斗篷味道睡着的小橘猫，是怎样一点点长大的。

---

## 1. 项目愿景

本项目的长期愿景是：

> 用 Markdown、Git 历史和持续记录，为马十一保存一份可以陪伴它一生的成长档案。

它可以是：

- 一份日记；
- 一份健康记录；
- 一份相册索引；
- 一份年度纪念册素材；
- 一份妈妈与小十一共同生活的时间胶囊。

如果未来导出为 PDF、打印成册，书名可以简单叫：

> **《马十一》**

首页固定文案：

> 欢迎回家，小十一。  
> 从今天开始，你姓马。

## 2. 基本设定

### 2.1 主角

- 大名：马十一
- 小名：十一 / 小十一
- 猫咪类型：小橘猫
- 回家日期：2026-06-26
- 回家时估计年龄：约两周龄
- 回家时状态：眼睛睁开，耳朵立起，会摇摇晃晃走路

### 2.2 名字由来

“十一”来自回家当天的农历五月十一。

“十一”与“拾遗”谐音，寓意：

> 不是遗弃，而是被拾起；不是偶然，而是回家。

### 2.3 妈妈

档案中的主人称呼统一为：**妈妈**。

不要写成爸爸。

这是一个重要设定，所有 AI 和后续维护者必须遵守。

---

## 3. 仓库定位

这个仓库不是代码仓库，虽然它使用 Git 维护。

它是一本书的原稿。

Git 的每一次 Commit，都是这本书翻过的一页。

因此：

- 不追求炫技；
- 不追求复杂工程；
- 不为了自动化牺牲可读性；
- 不让数据记录压过故事本身；
- 也不让故事记录取代健康观察。

本仓库的核心价值是：**长期、真实、可回溯。**

### 4.1 README.md

README 是给人看的首页。

它不需要太长。

它应当像一本书的封面和扉页，清楚传达：

- 这只猫是谁；
- 为什么记录；
- 这个项目从何时开始；
- 最新状态可以在哪里查看。

### 4.2 PROJECT_BIBLE.md

本文件是项目最高规范。

任何 AI 或维护者在修改仓库前，应先阅读本文件。

### 4.3 diary/DIARY.md

成长日记主文件。

所有 Day 记录优先追加到这里。

除非未来内容过长，否则不要按天拆分文件。

### 4.4 data/*.csv

CSV 用于保存结构化数据。

它们服务于未来绘图、统计、PDF 生成，但不应替代日记。

### 4.5 prompts/*.md

保存 AI 协作规范。

用于未来 ChatGPT、Codex 或其他 AI 接手时快速理解项目。

### 4.6 photos/

未来用于存放照片或照片索引。

如直接存大图导致仓库膨胀，可以只存压缩图或外链索引。

## 5. 更新原则

### 5.1 永远追加

成长记录原则上只追加，不覆盖。

历史 Day 不应随意重写。

如果必须修正错误，应在同一天记录中增加“修正说明”，而不是删除原文。

### 5.2 真实优先

所有故事必须来自妈妈实际描述、照片、视频或当天真实事件。

可以整理语言，但不能虚构事件。

可以温柔表达，但不能为了煽情制造不存在的情节。

### 5.3 数据与故事并重

每天记录应尽量包含：

- 日期；
- Day 编号；
- 体重；
- 奶量或饮食；
- 排尿；
- 排便；
- 精神状态；
- 今日故事；
- 今日观察；
- 今日成就。

如果某项未知，可以写“未记录”或“待补充”。

不要为了完整性编造数据。

### 5.4 平凡日子也值得记录

不是只有“第一次”才值得写。

如果某天只是：吃饱了、睡着了、平安无事，也可以记录。

对于曾经流浪的小奶猫来说，平安本身就是重要事件。

## 6. 文风规范

### 6.1 总体风格

文风应当：

- 温暖；
- 真实；
- 克制；
- 有生活感；
- 允许轻微幽默；
- 避免过度煽情。

可以写得有感情，但不要每天都写成散文诗。

### 6.2 叙述视角

默认使用第三人称或温柔记录体。

可使用“妈妈”作为人物称呼。

例如：

> 晚上，小十一喝完奶后仍然追着妈妈的手吸。妈妈重新冲了一点奶，它又喝下了 8ml。喝完以后，它不再急着吸，而是开始轻轻舔妈妈的手。

### 6.3 避免过度拟人

可以适度翻译行为，但不要把猫写得像真人说话过多。

允许偶尔使用“小十一翻译器”式的轻松段落，但正式日记中应以真实行为为主。

### 6.4 固定收尾

每个重要 Day 记录可以用这句作为收尾：

> 📖 今天的小十一，又长大了一点点。

项目长期固定结尾文案：

> 愿马十一永远不知道什么叫流浪。  
> 愿它只记得，自己一直都是马家的孩子。

## 7. Day 记录模板

推荐格式：

```markdown

### 今日概览

- 体重：
- 饮食：
- 排尿：
- 排便：
- 精神状态：

### 今日故事

（记录当天最值得回看的真实事件）

### 今日观察

（行为、健康、情绪、发育观察）

### 今日成就

> 📖 今天的小十一，又长大了一点点。
```

### 7.1 Day 编号

Day 0 为回家当天：2026-06-26。

之后每日递增。

如某天没有记录，不强行补写虚假内容；可在后续写“本日未记录”。

## 8. 数据记录规范

### 8.1 weight.csv

字段建议：

```csv
date,day,weight_g,measurement_time,notes
```

说明：

- `date`：日期，格式 YYYY-MM-DD；
- `day`：Day 编号；
- `weight_g`：体重，单位克；
- `measurement_time`：称重时间，如 morning / after_feeding / unknown；
- `notes`：备注，如“饭后称重”。

### 8.2 milk.csv

字段建议：

```csv
date,day,time,amount_ml,notes
```

不要只记录总量；奶猫阶段每顿记录更有价值。

如果只有总量，可在 time 写 `total`。

### 8.3 health.csv

字段建议：

```csv
date,day,pee,poop,energy,notes
```

示例：

```csv
2026-06-27,1,yes,no,good,排尿正常，暂未排便，继续观察
```

### 8.4 医疗记录

未来如出现驱虫、疫苗、就医、用药，应建立或更新：

```text
diary/medical.md
```

医疗记录必须客观，不夸张，不自行替代兽医诊断。

## 9. 医疗与健康记录原则

本项目可以记录健康观察，但不能把 AI 观察写成正式诊断。

写法应保持谨慎：

- “观察到……”
- “目前表现为……”
- “建议继续观察……”
- “如持续出现，应咨询兽医……”

避免写：

- “确定没事”；
- “一定是……”；
- “不用看医生”；
- “这是病 / 不是病”的绝对判断。

对于奶猫阶段，重点关注：

- 体重趋势；
- 食欲；
- 排尿；
- 排便；
- 精神状态；
- 体温与保暖；
- 呼吸状态；
- 是否呛奶；
- 是否腹泻。

照片索引可记录：

视频可不直接存仓库，只记录索引和描述。

## 12. Git 与 Commit 规范

Commit 信息应简洁、可读、有纪念意义。

推荐格式：

```text
docs(day-001): 第一次正式称重
docs(day-002): 第一次成功排便
data(weight): update day 002 weight
chore(repo): initialize project structure
```

对于每日成长更新，推荐：

```text
docs(day-XXX): 简短中文主题
```

不要使用无意义信息：

- update
- fix
- aaa
- test

测试文件完成后可以删除，但保留 Git 历史即可。

## 13. AI 协作规范

### 13.1 ChatGPT 职责

ChatGPT 负责：

- 整理妈妈提供的每日信息；
- 写成长日记；
- 进行行为观察；
- 提醒健康风险；
- 更新 Markdown；
- 保持文风统一。

ChatGPT 不应：

- 虚构当天没有发生的事情；
- 擅自改写历史 Day；
- 把不确定医学判断写成确定诊断；
- 忽略妈妈明确提供的事实。

### 13.2 Codex 职责

Codex 如参与，主要负责：

- 文件结构维护；
- 自动化脚本；
- CSV 处理；
- PDF 导出；
- GitHub Actions；
- 格式检查。

Codex 不应主动改写成长日记文风。

### 13.3 未来其他 AI

任何未来 AI 接手本项目时，必须先阅读：

1. `PROJECT_BIBLE.md`
2. `README.md`
3. `diary/DIARY.md`
4. `data/*.csv`

再进行任何更新。

## 14. 禁止事项

本项目禁止：

- 为了感人而虚构事件；
- 随意删除历史记录；
- 把妈妈称为爸爸；
- 把医疗建议写成诊断；
- 用夸张标题污染日记；
- 大量使用无意义 emoji；
- 把项目变成纯数据表，失去故事；
- 把项目变成纯抒情文，失去事实。

## 17. 最终原则

如果未来维护者只记住一条规则，请记住这一条：

> 记录真实发生的事，保留未来还会想看的细节。

《马十一》不是为了写得漂亮。

它是为了很多年以后，妈妈重新打开时，能想起那只小小的橘猫曾经怎样来到家里，怎样喝奶，怎样踩奶，怎样一点点长大。
